# README and v1.1.2 Release Design

## Goal

Rewrite the repository README around the product's real value: converting complex images into saveable Rockstar emblems when fast online logo-oriented tools are not suitable. Publish the verified Windows package as GitHub Release v1.1.2 after the README is updated.

## Audience

The README must work for three readers without giving any of them a misleading first impression:

- Emblem creators who want to convert photos, illustrations, anime artwork, or other complex images.
- Technical readers who want to understand why the converter can preserve more structure and detail.
- Recruiters and engineers evaluating the search, GPU, native application, and reverse-engineering work.

## Product Positioning

Name Emblem Helper directly and describe the tradeoff fairly:

- Emblem Helper is an extremely fast online tool that requires no installation and is well suited to structurally simple, logo-like images.
- GTA Emblem Maker complements it in the opposite direction: it requires a Windows download, an NVIDIA CUDA-capable GPU, and substantially more processing time, but it is designed for complex images where a simple instant conversion is insufficient.
- Do not claim that one tool is universally better. Explain that they serve different source images and different quality/time priorities.

The primary claim is that GTA Emblem Maker spends compute on constrained layered reconstruction so users can attempt complex image-to-emblem conversions that are outside the intended strength of instant logo-oriented tools.

## README Structure

### 1. Hero

Keep one restrained dark/light banner. Lead with a concrete statement such as "A quality-first GTA emblem converter for complex images." The supporting sentence should mention photos and illustrations, the time-for-detail tradeoff, and Rockstar-compatible output. Do not lead with WPF, CUDA, transparency, or pipeline count.

### 2. Emblem Helper Comparison

Add a compact comparison table covering source-image fit, execution model, speed, installation, hardware, and intended use. Avoid unsupported benchmark numbers and absolute quality claims.

### 3. Product Outcomes

Replace the current generic feature grid with four outcome-oriented cards:

- Made for complex images.
- Quality over instant results.
- Budget-aware detail fitting.
- Save-ready Rockstar output.

Each card must explain a user-visible result. GPU support, native UI, transparency, and pipeline count must not appear as headline selling points.

### 4. Why It Works

Explain the technical basis in plain English before naming implementation details:

- Model the source as a layered primitive reconstruction under Rockstar's code-size constraint.
- Search position, size, rotation, color, alpha, and primitive family instead of tracing only a simple outline.
- Use beam search to reduce early greedy mistakes.
- Combine pixel error with LPIPS perceptual reranking or native edge-detail reranking where configured.
- Mix rotated primitives with verified official Rockstar catalog shapes.
- Keep export cost in the search loop so visual improvement does not produce an unusable payload.

These are explanations for the complex-image capability, not a detached algorithm-name list.

### 5. Engineering Challenges

Describe the engineering required to make the quality-first approach practical:

- Batch candidate generation and scoring on CUDA and keep the scorer alive across layers.
- Reduce repeated CPU/GPU transfers and expose progress without restarting the search process.
- Integrate ONNX Runtime DirectML for the LPIPS path and a native edge-detail path for catalog fitting.
- Preserve cancellation, previews, deterministic run artifacts, traces, and payload inspection in the native application.
- Reverse-engineer and verify Rockstar's actual catalog geometry, cleanup precision, SVG path serialization, and save behavior instead of assuming catalog declarations can be submitted directly.

CUDA belongs here and in requirements. Transparent input handling belongs in supported-input correctness, not in the product pitch.

### 6. Usage and Pipelines

Provide a short four-step workflow: download, open an image, run one or all profiles, compare results and copy the import code. Retain a factual three-profile table for Beam Clean, Perceptual AlexNet 224, and Official Catalog Quality. Do not label any profile as universally best.

### 7. Requirements and Tradeoffs

State the costs clearly:

- Windows x64 and .NET Framework 4.8.
- NVIDIA CUDA-capable GPU.
- Large or difficult images can take a long time.
- The output is still an approximation constrained by Rockstar's supported primitives and payload size.
- Transparent inputs are supported as expected.

### 8. Examples

Include an `Examples` section that says real comparisons will be added later. Mention the planned categories: cat photos, anime illustrations, and optimization/conversion comparisons. Do not add fake screenshots, stock images, or empty image frames.

### 9. Build and Project Status

Keep concise build instructions, package contents, unofficial-project disclaimer, and third-party license expectations. Remove decorative sections that do not add product or engineering information.

## Visual Direction

Retain restrained dark/light SVG support, the banner, and simple dividers. Rewrite the feature SVG around product outcomes. Decorative graphics may support hierarchy but must not replace explanation. The README should read as technical product documentation first and a designed landing page second.

## v1.1.2 Release

After the README implementation is verified:

1. Commit and push the final README and SVG changes to `main`.
2. Tag that final `main` commit as `v1.1.2`.
3. Create a public, non-prerelease GitHub Release named `GTA Emblem Maker v1.1.2`.
4. Upload `release/GTAEmblemMaker-v1.1.2.zip`.
5. Link the README quick-start download to the v1.1.2 release asset.

Release notes must cover:

- Rockstar-compatible official catalog export geometry, cleanup precision, and serialized SVG path fixes.
- The retained decimal 1,270,000 generated-code budget; 1,280,000 was tested and rejected by Rockstar.
- Removal of the misleading Best Quality profile.
- Beam Clean as the default and three packaged profiles.

## Acceptance Criteria

- A new reader can identify the complex-image use case in the first screen.
- Emblem Helper is named and compared fairly without claiming universal superiority.
- Product outcomes are separated from algorithms, engineering methods, baseline correctness, and requirements.
- The README explains enough real search and systems work to be useful to technical readers and recruiters.
- No fake examples are included.
- The quick-start link points to v1.1.2.
- The published release asset matches the locally verified ZIP and contains exactly three profiles.
- The latest GitHub Release is v1.1.2 and is not a draft or prerelease.
