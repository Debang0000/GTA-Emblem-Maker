# GTA Emblem Maker v1.1.2 Catalog Export Design

## Goal

Make `Official Catalog Quality` payloads acceptable to Rockstar without changing the v1.1.1 fitting schedule, selected layers, preview quality, or CUDA performance. Keep every production profile at the reliable `1,250,000`-character budget.

## Evidence

The original v1.1.1 payload and a shortened version both failed with Rockstar's generic save error. The same result saved successfully after removing only the 392 layers that used the 11 new Catalog slugs. This isolates the regression to Catalog export compatibility rather than clipboard handling or total payload length.

The v1.1.1 Catalog definitions preserve recovered high-precision geometry for fitting. Their exported path strings and intrinsic dimensions are not the canonical values used by Rockstar's current editor. For example, v1.1.1 exports `rounds/08` as `211.872 x 300.001` and `rounds/18` as `262.57 x 300`, while Rockstar defines them as `212 x 300` and `263 x 300`.

## Chosen Approach

Keep the existing high-precision Catalog definitions for fitting, CUDA atlas generation, candidate selection, and working previews. Add a separate canonical export definition for each of the nine curve and two round slugs. `RockstarExporter` and payload preview rendering will use the canonical export path and dimensions only when serializing a Catalog `ShapeState`.

The export transform will be recomputed against the canonical intrinsic dimensions while preserving the fitted center, width, height, rotation, color, and opacity. This keeps the visible geometry unchanged while making `svgData` and `layerData` agree with Rockstar's official shape metadata.

Do not change the legacy ellipse, rectangle, triangle, or line export path. Do not change Catalog search, scorer, masks, seeds, layer schedule, reranking, or sub-four-axis fitting behavior.

## Budget Change

Set the stage budget to decimal `1,250,000` in every production profile:

- Best Quality
- Beam Clean
- Perceptual
- Official Catalog Quality

The budget change is independent of Catalog compatibility. Automated checks will verify exact profile values, but only a live Rockstar save can establish the server's accepted upper boundary.

## Tests

Add failing checks before production edits that require all 11 Catalog exports to use canonical Rockstar paths and intrinsic dimensions. Include representative anisotropic and sub-four-axis states so the exported visual bounds remain equivalent.

Keep the existing legacy exporter goldens byte-for-byte unchanged. Keep the existing Catalog fitting and trace checks unchanged. Add checks that every production profile uses `1,250,000`.

Run the CUDA scorer build, Release solution build, and the complete checks executable. Do not package or publish v1.1.2 in this phase.

## Acceptance

Provide the locally built application for a live test. Acceptance requires an unmodified Catalog payload to save successfully through `/emblems/save`. If canonical export still fails, preserve this branch evidence and evaluate the broader option of changing fitting definitions; do not silently remove Catalog layers.

## Scope Boundary

This phase changes source, tests, profile budgets, and version metadata to `1.1.2`. It does not create a release archive, tag, GitHub release, or push.
